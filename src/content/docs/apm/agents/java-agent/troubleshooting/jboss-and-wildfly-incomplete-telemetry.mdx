## Problem

You are getting incomplete telemetry from an app using JBoss EAP 7.4 (or up) or Wildfly 23 (or up). 
This could be in the form of web requests being translated into non-web transactions or your dispatcher name is incorrect.

If you set agent logs using `Finest` log level and find `NoClassDefFoundError` stack traces, this most likely means this is a problem with how
class-loading works with JBoss EAP/Wildfly. 

The agent weaver can instrument a specific class in an app using JBoss but the modified classes could use packages and 
modules not recognized by JBoss EAP/Wildfly. The result here is the ClassLoader cannot find specific classes throwing a `NoClassDefFoundError`. While 
your app will still work, it means the data sent by the agent will be incorrect or incomplete. Since agent version `8.6`, the java agent can make JBoss 
recognize certain packages so Wildfly/JBoss EAP instrumentation will work out of the boss. However this doesn't guarantee `NoClassDefFoundError`s will not
come up again.

## Solution

Set the agent log level to `Finest` and in your agent log file, look for stack traces containing `NoClassDefFoundError`.

Since the class `java.util.logging.Level` is causing the error, we have to set a system property called `jboss.modules.system.pkgs` to make JBoss recognize all classes in the `java.util.logging` package.

In this case you would set the system property using a comma seperated list of Java packages you want JBoss to recognize.

For example: `Djboss.modules.system.pkgs=java.util.logging,javax.management`. 

It is recommended though to contact [support](https://support.newrelic.com/s/) if this problem is encountered. This is because if `NoClassDefFoundError` is found in the agent logs,
it is a class used by our agent. So being notified of this will open the door for later releases to provide a patch to make JBoss recognize more packages.
